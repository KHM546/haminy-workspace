# AIGetDone 功能优化建议文档

更新时间：2026-04-24

## 1. 这份文档要解决什么问题

这份文档不是重新做一遍竞品分析，而是把两件事合起来：

- `AIGetDone` 现有代码和功能底子到底已经到哪一步了
- 结合上一份 [竞品分析文档.md](E:\haminy-workspace\竞品分析文档.md)，接下来最值得补哪些功能

目标只有一个：

**把建议收敛成“基于现状能落地”的产品优化清单，而不是空泛讲战略。**

## 2. 先给结论

如果结合当前代码现状来看，`AIGetDone` 最值得优先做的，不是再加更多模型选项，而是把已经有的底层能力继续打磨成 7 个用户能直接理解、直接能用的功能结果：

1. `现有任务模板中心 / AI Agent 机器人体系优化`
2. `执行时间线 + 关键动作确认`
3. `渠道消息转任务 + 结果回流`
4. `团队共享文件夹`
5. `UI 界面美化`
6. `程序响应流畅度优化`
7. `异常中心 / 驾驶舱`

原因很简单：

- 你们现在已经有任务、Worker、多 Runner、渠道接入、角色市场、工作区能力这些底子
- 但这些能力还比较像“技术能力集合”
- 用户能感知到的“产品结果层”还不够强

换句话说，当前最大的机会不是“补底层”，而是“补产品封装”。

## 4. 当前最明显的产品缺口

结合代码和竞品对照，我认为当前最明显的缺口有 6 个。

### 4.1 有 Agent 体系，但缺“更好用的任务入口”

你们已经有 Worker、Market、Skills、Runner、Workspace、Channels，但普通用户仍然要自己想：

- 该选哪个 Agent
- 该怎么写 prompt
- 该用哪个模型
- 输出应该长什么样

这就是为什么 `SellClaw` 那种“直接卖 listing、库存、广告”的表达更强。

### 4.2 有状态事件，但缺“执行过程感”

当前 Runner 状态更多是短文案和 SSE 事件，没有形成让用户放心的过程 UI。

这和 `Manus` 的差距不在底层能力，而在：

- 用户是否看得到每一步
- 用户是否能在关键写操作前确认

### 4.3 有渠道接入，但缺“渠道任务化”

当前渠道设置已经很完整，但产品价值还更偏“配置好了某个入口”，而不是：

- 群里 @AI 自动建任务
- 结果回原线程
- 失败时自动给原因和下一步建议

### 4.4 有角色市场，但缺“团队共享文件夹”

当前已经有 Role Market，但更像“角色安装”。
但团队协作层还不够实用，尤其缺少一个大家都能理解的共享单位。
相比“团队共享空间”这种偏抽象表达，当前更适合落成 `团队共享文件夹`：

- 团队把常用模板放进去
- 团队把常用 Agent 放进去
- 团队把常用结果范本放进去
- 团队按项目或部门整理文件夹

### 4.5 UI 完成度还不够强

从现有界面结构看，功能已经不少，但产品外观还有几个明显问题：

- 有些区域更像管理后台，而不是成熟桌面产品
- 信息密度高，但层级感不够强
- 用户第一次打开时，不容易快速理解“哪里最重要”

这会直接影响对产品专业度的判断。

### 4.6 程序响应流畅度还需要继续优化

当前产品功能链已经比较长，涉及聊天、任务、渠道、文件、市场、设置等多个区域。
这种产品如果切换、加载、刷新、渲染做得不够顺，会直接让用户觉得“功能很多但不跟手”。

这块虽然不是单一功能，但对整体口碑影响很大。

## 5. 优先级最高的功能优化建议

下面这几条建议，我按“投入产出比 + 复用现有代码能力”排序。

### 5.1 P0：现有任务模板中心 / AI Agent 机器人体系优化

**为什么先做**

你已经说明了，项目里其实已经有任务模板中心，而且本质上就是一批培训好的 AI Agent 机器人。

所以这里不应该再写成“新建模板中心”，而应该写成：

**把现有 Agent 机器人体系变得更像成熟产品。**

**当前依据**

- Worker 已有：[route.ts](E:\haminy-workspace\aigetdone_client\src\app\api\workers\route.ts:25)
- Worker 编辑已存在：[WorkerRoleEditor.tsx](E:\haminy-workspace\aigetdone_client\src\components\workers\WorkerRoleEditor.tsx:35)
- 角色市场已存在：[RoleMarketBrowser.tsx](E:\haminy-workspace\aigetdone_client\src\components\market\RoleMarketBrowser.tsx:130)

**当前缺什么**

- 机器人虽然已有，但“怎么选”“适合干嘛”“输入什么最容易成功”还不够清楚
- Agent 卡片更像角色展示，还不够像任务入口
- 不同 Agent 之间的差异没有被包装成清晰结果
- 缺少“推荐使用场景”和“成功输出样例”

**建议怎么做**

不要先扩模板平台，先优化现有机器人中心的使用体验。

建议直接补这 6 件事：

- `Agent 卡片增加适用场景`
  例如“适合：竞品分析 / 周报 / 会议纪要 / 客户回复”
- `Agent 卡片增加输入要求`
  告诉用户要给什么材料，成功率最高
- `Agent 卡片增加输出示例`
  让用户一眼知道结果会长什么样
- `Agent 卡片增加推荐模型或推荐模式`
  降低选择成本
- `按任务分类展示 Agent`
  不只按角色名展示，而是按“写作、分析、代码、运营、沟通”分类
- `最近使用 / 常用 Agent 置顶`
  提升二次使用效率

如果需要补充高频模板，建议只补 6 到 8 个最常用场景：

- 竞品分析
- 群聊总结
- 会议纪要转行动项
- 项目周报
- 客户回复草稿
- 代码审查
- 发布说明生成
- 渠道消息分派

**建议落点**

- 优先改 [WorkerGrid.tsx](E:\haminy-workspace\aigetdone_client\src\components\workers\WorkerGrid.tsx:7)
- 优先改 [WorkerRoleEditor.tsx](E:\haminy-workspace\aigetdone_client\src\components\workers\WorkerRoleEditor.tsx:35)
- 优先改 [RoleMarketBrowser.tsx](E:\haminy-workspace\aigetdone_client\src\components\market\RoleMarketBrowser.tsx:130)

### 5.2 P0：执行时间线 + 关键动作确认

**为什么先做**

这是和 `Manus` 最直接的差异化学习点，而且你明确希望这里写得更具体。

**当前依据**

- Runner 路由：[agent-runner-gateway.ts](E:\haminy-workspace\aigetdone_client\src\lib\agent-runner-gateway.ts:731)
- Codex app-server 事件源：[agent-runner-gateway.ts](E:\haminy-workspace\aigetdone_client\src\lib\agent-runner-gateway.ts:494)
- Direct chat 事件源：[agent-runner-gateway.ts](E:\haminy-workspace\aigetdone_client\src\lib\agent-runner-gateway.ts:615)
- 群组执行：[group-codex-executor.ts](E:\haminy-workspace\aigetdone_client\src\lib\group-codex-executor.ts:26)
- 当前状态文案：[runner-status.ts](E:\haminy-workspace\aigetdone_client\src\lib\runner-status.ts:8)

**当前缺什么**

- 只有短状态，没有完整 timeline
- tool_use、tool_result、permission_request 没被整理成稳定 UI
- 用户不知道 AI 当前到底在“读”“想”“调用工具”“等待确认”还是“快完成了”

**建议怎么做**

第一版时间线建议拆成 6 个固定阶段，不要让每个模型自由发挥：

1. `接收任务`
   显示：任务标题、发起来源、当前使用的 Agent
2. `理解与拆解`
   显示：AI 识别出的任务目标、预计步骤数
3. `读取上下文`
   显示：读取了哪些文件、会话、渠道消息、附件
4. `执行动作`
   显示：调用了什么工具、做了什么动作、动作结果是否成功
5. `等待确认`
   显示：为什么要确认、确认后会执行什么、是否属于写操作/外发动作
6. `生成结果`
   显示：结果摘要、失败原因、下一步建议

界面上建议固定 4 种状态样式：

- `进行中`
- `已完成`
- `等待确认`
- `失败`

关键确认点建议固定收口成 4 类：

- 写文件
- 发消息 / 回渠道
- 调外部服务
- 影响已有任务结果的覆盖操作

用户能从时间线里直接做的动作建议只有 4 个：

- 展开看详情
- 确认继续
- 停止执行
- 失败后重试当前步骤

**建议落点**

- 新增 `src/lib/execution-timeline.ts`
- 新增 `src/components/chat/ExecutionTimeline.tsx`
- 新增 `src/components/chat/ActionConfirmCard.tsx`

### 5.4 P1：团队共享文件夹

**为什么做**

这是把 `AIGetDone` 从“个人桌面 AI 客户端”往“团队可协作工具”推进的最容易理解的一步。
相比“团队共享空间”，`团队共享文件夹` 更具体，也更容易让用户马上明白怎么用。

**当前依据**

- Worker 市场：[RoleMarketBrowser.tsx](E:\haminy-workspace\aigetdone_client\src\components\market\RoleMarketBrowser.tsx:130)
- Worker 体系：[WorkerGrid.tsx](E:\haminy-workspace\aigetdone_client\src\components\workers\WorkerGrid.tsx:7)

**当前缺什么**

- 团队还没有一个明确的共享单位来放 Agent、模板、常用文件和结果
- 好用内容容易留在个人侧，难以沉淀
- 当前更像“个人使用后再手工转发”，不是团队共用

**建议怎么做**

第一版不要做复杂协作系统，只做一个清晰的共享文件夹机制：

- `团队共享文件夹`
  可按部门、项目、用途建文件夹
- `共享 Agent`
  某个 Agent 可被保存到团队文件夹
- `共享模板`
  常用任务模板可放进团队文件夹
- `共享结果`
  某次生成的报告、话术、周报可一键保存进去
- `共享素材`
  常用输入文件、附件、链接集合可归档

### 5.5 P1：UI 界面美化

**为什么做**

功能再多，如果界面看起来像“功能集合页”，用户会天然觉得产品还不够成熟。

**当前问题**

- 部分页面信息层级不够清晰
- 有些区域更像后台配置页，产品感不够强
- 重点区域、次要区域、辅助说明的视觉分层不够稳定

**建议怎么做**

优先改 5 个最影响观感的区域：

- `Worker / Agent 列表页`
  让卡片更像“可直接开工的机器人”，不是普通数据卡片
- `聊天主界面`
  顶部状态、消息区、时间线区、输入区要更有层级
- `任务列表`
  让状态、优先级、最近更新时间更好扫读
- `渠道设置页`
  把“已连接”“待配置”“有风险”做成更明确的视觉状态
- `市场页`
  提升商品化、成品化感觉，不要太像内部管理页

UI 美化建议重点抓 4 个方向：

- 统一卡片层级和圆角、边框、阴影规则
- 强化重点按钮、状态标签、空态页设计
- 减少一屏过多纯文字说明，增加结构化模块
- 增强首屏可理解性，让用户 5 秒内知道该点哪里

### 5.6 P1：程序响应流畅度优化

**为什么做**

这是最容易影响口碑但最容易被忽略的一块。
用户不一定会说“你们少了哪个功能”，但一定会感觉“这个程序顺不顺手”。

**建议优先优化的点**

- `聊天发送后的首字响应速度`
  尽量缩短“点发送到看到第一段输出”的等待感
- `页面切换速度`
  Worker、设置、市场、聊天之间切换不要有明显卡顿
- `大列表渲染`
  Worker 列表、市场列表、会话列表避免一次性全量重渲染
- `任务刷新`
  任务列表更新要平滑，避免闪烁
- `渠道设置保存`
  保存后即时反馈，不要让用户怀疑是否成功
- `加载占位`
  所有 300ms 以上的加载都建议有稳定 skeleton 或 loading 状态

**建议怎么做**

第一阶段先做可感知优化，不先做重度底层重构：

- 给关键区域补 skeleton
- 减少重复请求和不必要刷新
- 大列表加分页、窗口化或延迟渲染
- 将“阻塞用户操作”的串行请求改成并行或分步加载
- 把成功、失败、处理中状态做得更即时

### 5.7 P2：异常中心 / 驾驶舱

**为什么做**

这条建议主要来自 `领星 ERP`。你们现在页面更多是“配置页”和“功能页”，还缺一个明确告诉用户“今天该处理什么”的运营入口。

**建议怎么做**

第一版异常卡片只收 5 类：

- 任务执行失败
- 索引过期
- 某渠道待处理
- 某模板输出异常
- 某 Runner 配置失效

每张卡片都能一键发起 AI 处理。

## 6. 不建议现在做的事

为了避免计划过大，这几件事我反而建议先别做：

- 不要先做完整模板市场
- 不要先做知识权限治理大系统
- 不要先做行业 ERP 深连接大全
- 不要先做特别复杂的模型评分和自动调度系统
- 不要先扩更多设置页

原因是：

- 当前最大短板不是“底层能力不够多”
- 而是“能力没有变成清晰的产品结果”

## 7. 建议的落地顺序

### 第一阶段：2 周内最值得做

- `现有 Agent / 模板中心优化`
- `执行时间线 MVP`
- `UI 首屏和主流程美化`

### 第二阶段：再往前推 2 到 4 周

- `渠道消息转任务`
- `结果回流`
- `团队共享文件夹`
- `程序响应流畅度优化`

### 第三阶段：验证团队价值

- `共享模板`
- `共享 Agent`
- `异常中心`

## 8. 一句话判断

如果只看代码底座，`AIGetDone` 现在已经具备往“AI 员工工作台”走的基础能力。

真正限制产品继续往前走的，不是底层缺少模型、索引、渠道或 Worker，而是：

**这些能力还没有被包装成用户一看就懂、团队一用就有结果的功能层。**

所以接下来最值得做的，不是“再增加能力种类”，而是把已有能力收敛成：

- 任务入口
- 执行过程感
- 团队共享文件夹
- 结果回流
- 更好的界面完成度
- 更顺滑的操作体验
